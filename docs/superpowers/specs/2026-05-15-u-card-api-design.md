# U Card API Integration Design

## Goal

Build the first usable U card integration for the existing Claw800/Nexa app:

- Admin can configure the upstream U card platform credentials from `https://b.alipay.bot/api`.
- Admin can generate a local developer RSA key pair, copy the customer public key to the upstream platform, then paste the upstream platform public key back into Claw800.
- Mobile users sign in only through Nexa third-party login.
- Mobile users see two tabs: `所有卡` and `我的卡`.
- After a user applies for a card, that card appears in `我的卡`.
- Nexa third-party payment is the only payment path for initial card funding or card application payment.

This spec covers the minimum viable closed loop. Card sensitive-info viewing, freeze/unfreeze, balance modification, transaction history, and recharge management remain out of scope for this first pass.

## Existing Context

The repo already has:

- Nexa OAuth and payment helpers in `src/nexa-pay.js`.
- Nexa session exchange through `/api/nexa/tip/session`.
- App-specific Nexa session cookies for escrow, mining, chat, and games in `src/server.js`.
- A current `/u-card-query/` page that queries static supported card BINs by payment platform.
- Admin U card management for manual platform/card metadata.
- A SQLite `settings` table suitable for storing integration configuration.

The upstream U card API uses:

- Base URL: `https://b.alipay.bot/backend`
- Request headers: `x-app-id`, `x-timestamp`, `x-nonce`, `x-signature`
- Signature formula: `sha256(appId + "." + timestamp + "." + nonce + "." + JSON.stringify(body) + "." + apiKey)`
- Core endpoints:
  - `POST /open-api/cards/products`
  - `POST /open-api/cardholders`
  - `POST /open-api/cardholders/query`
  - `POST /open-api/cards/open`
  - `POST /open-api/cards/query`
  - `POST /open-api/cards/list`

## Approach

Use a backend proxy integration. The browser never receives upstream API keys, private keys, or upstream platform public keys.

The primary public mobile U card entry must be `https://claw800.com/u`, served by route `/u`. The existing `/u-card-query/` route can remain as a compatibility redirect to `/u`, but the product path, Nexa redirect URI, and shared links should use `/u`.

The U card entry must also be listed on the `游戏与工具` page as the final item, linking to `/u`.

Recommended initial scope:

1. Admin config and key management.
2. Upstream product list retrieval.
3. Nexa login session for the U card page.
4. Cardholder creation per Nexa user.
5. Card application.
6. User card list from local records plus upstream query refresh.

## Admin Configuration

Add a `U 卡接口` configuration panel in the existing admin UI.

Fields:

- U card API base URL, default `https://b.alipay.bot/backend`.
- APP ID, pasted by admin.
- API Key, pasted by admin.
- Callback secret, pasted by admin when available.
- Developer private key, generated by Claw800 and stored server-side.
- Customer public key, generated from the private key and shown for copying to upstream.
- Upstream platform public key, pasted back after upstream generates it.

Actions:

- `生成密钥对`: creates a new RSA key pair and stores the private key and public key in `settings`.
- `测试连接`: calls a harmless signed upstream endpoint, preferably `cards/products` with a small JSON body.
- `保存配置`: persists values, masking secrets on read.

Admin API shape:

- `GET /api/admin/u-card/integration-config`
- `PUT /api/admin/u-card/integration-config`
- `POST /api/admin/u-card/generate-keypair`
- `POST /api/admin/u-card/test-connection`

## Data Model

Add local persistence so the user experience does not depend entirely on live upstream response shape.

`u_card_user_sessions` is not needed because the existing cookie pattern can be reused. Use a new cookie name such as `u_card_session`.

Add `u_card_cardholders`:

- `id`
- `user_id`
- `openid`
- `cardholder_id`
- name/contact fields submitted upstream
- `created_at`
- `updated_at`

Add `u_card_applications`:

- `id`
- `openid`
- `merchant_order_no`
- `cardholder_id`
- `product_code`
- `fee_amount`
- `currency`
- `init_balance`
- `platform_card_no`
- `status`
- `upstream_payload`
- `created_at`
- `updated_at`

Local records should be upserted by `merchant_order_no` and by `platform_card_no` when available.

## Mobile User Flow

The mobile page at `/u` becomes:

- Header with login state and Nexa login button.
- Segmented control: `所有卡` / `我的卡`.
- `所有卡`: calls backend for upstream products and renders card products.
- Product card action: `申请卡`.
- Application modal: collects only required minimum fields for MVP:
  - first name
  - last name
  - email optional
  - phone optional
  - initial balance optional
- Backend creates or reuses the user cardholder, then calls upstream `cards/open`.
- On success or pending result, local application is saved.
- `我的卡`: reads local applications for this Nexa `openid`, optionally refreshing each known card through `cards/query`.

If the user is not logged in, both `申请卡` and `我的卡` prompt Nexa login through the same custom protocol pattern already used by escrow and games:

`nexaauth://oauth/authorize?apikey=<NEXA_API_KEY>&redirect_uri=<current-page>`

## Payment Flow

Use Nexa third-party payment only.

For MVP, card application should support two payment-compatible paths:

- If upstream `cards/open` directly charges the U card merchant account, the frontend still requires a Nexa payment before calling open-card when the configured product requires prepayment.
- If the business decides to fund cards from the platform balance, the MVP can create the upstream card immediately and record the charge as an internal operation.

Implementation should expose a backend switch in settings:

- `u_card_payment_mode = "nexa_before_open" | "platform_balance"`

Default to `platform_balance` until the exact commercial flow is confirmed, because the upstream card API docs show card opening charges the merchant account via `feeAmount` rather than creating a public checkout URL.

When `nexa_before_open` is enabled, reuse existing Nexa payment creation/query helpers and store the pending order against `u_card_applications` before calling upstream `cards/open`.

## Backend API

Public/mobile endpoints:

- `GET /api/u-card/session`
- `POST /api/u-card/session`
- `POST /api/u-card/session/logout`
- `GET /api/u-card/products`
- `GET /api/u-card/my-cards`
- `POST /api/u-card/apply`
- `POST /api/u-card/applications/:merchantOrderNo/refresh`

Internal helper behavior:

- Normalize upstream errors into `{ ok: false, error }`.
- Never return API keys, private keys, platform public key, or raw sensitive card data to the browser.
- Store full upstream response JSON in local DB for debugging, but only expose safe fields publicly.

## Security

- Keep APP ID public only where needed for signed backend requests; do not expose API Key.
- Store private key and API Key only in `settings`.
- Admin config reads should return booleans such as `hasApiKey` and masked values, not raw secrets.
- Validate all money fields with two-decimal precision.
- Generate idempotent `merchantOrderNo` values server-side.
- Do not implement `/cards/secure-info` in the MVP.
- Verify upstream callbacks in a later pass using `x-upal-biz-type`, `x-upal-biz-no`, `x-upal-timestamp`, `x-upal-signature`, and callback secret.

## Error Handling

Mobile page states:

- Not logged in: show Nexa login call to action.
- Not configured: show service unavailable message.
- Upstream failure: show concise retryable message.
- Application pending: show pending card in `我的卡`.
- Application active: show active card summary.

Admin page states:

- Missing APP ID/API Key.
- Customer public key generated but upstream platform public key missing.
- Test connection success/failure.

## Testing

Add focused tests:

- U card upstream signature builder creates exact `x-signature`.
- Admin config masks secrets and persists generated public/private keys.
- Public product endpoint rejects missing config cleanly.
- Nexa-authenticated user can apply for a card and sees it in `我的卡`.
- Apply flow is idempotent for retries.
- Mobile page contains `所有卡` and `我的卡` controls and does not expose secrets.

## Open Questions

- Does upstream require RSA encryption with the generated customer/platform keys for any fields, or are those keys only for platform-side setup and future sensitive data exchange?
- Should card application require Nexa payment before opening, or should the platform merchant balance pay upstream immediately?
- Which exact product fields from `cards/products` should be displayed to users once real credentials are available?
